d-server: fail closed under overload instead of exhausting threads

The d-server built its gRPC transport with all defaults, so it had no way
to refuse work it could not serve. Offered more concurrent requests than
the host could handle, the excess queued without bound and contended for
runtime threads rather than being told to back off, and a stuck handler
could pin its slot indefinitely. Under sustained overload the server
degraded into thread-pool exhaustion — the opposite of the intended
fail-closed-under-pressure behaviour.

Configure the transport with a server-wide concurrency bound plus load
shedding, so once the admission limit is reached across all connections
the excess is refused immediately with a retryable RESOURCE_EXHAUSTED
"busy" status instead of piling up. Add a per-request timeout so a hung
handler is cut loose rather than holding its slot forever, and bound the
HTTP/2 inbound-stream and TCP posture. The admission limit is
operator-tunable (CLI flags and the AdmissionControl config) so it can be
matched to the backing device's useful queue depth — shallow for an HDD
spindle, deep for SSD/NVMe — rather than a fixed constant.

Also makes the custodian durability-metrics simulation test deterministic
under parallel execution. A permissive process-global tracing default is
now installed once so the per-callsite interest cache cannot be poisoned
to "never" by a non-capturing test that races to the callsite first; that
race previously left the emission capture empty and flaked the assertion.
