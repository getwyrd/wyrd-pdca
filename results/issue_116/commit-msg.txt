test(dst): network DST over the real gRPC ChunkStore on madsim

The gRPC ChunkStore data path landed across M2.1-M2.5 (proto service,
client + D-server service, d-server discovery, parallel fan-out write,
any-k read), but nothing exercised that wire code under network faults:
testkit had Clock and Disk seams yet no network seam, and the DST suite
only drove the in-process commit protocol. A drop, delay, partition, or
on-the-wire corruption could regress silently.

Grow a network seam in testkit and run the *real* GrpcChunkStore over
madsim's deterministic simulated network, so every put/get is a gRPC
round-trip the simulator can fault and replay from its seed. This is
proposal 0004's primary approach, not the in-sim-fake fallback: the same
tonic client and service code is cfg-aliased to madsim-tonic under
--cfg madsim, so the build under test is the shipping wire code, not a
stand-in.

The new campaign asserts the five Tier-1 properties from proposal 0004
over that wire: parallel-write durability, k-of-n read with dropped
fetches, re-read-on-corruption, fail-closed partial write, and the
M0/M1 commit suite (concurrent-writer-one-wins) re-run over the gRPC
store to prove the trait seam is real. Faulted-link selection is drawn
from the run seed, so a bug-finding seed replays the same faults as a
permanent regression test (ADR-0009); none surfaced over the 50-seed
sweep. Determinism holds despite the new fan-out parallelism, since
completion ordering is decided by madsim's seeded scheduler.

Scope is proposal 0004 suggested PR step 6 only; Tier-2 container
integration and the throughput benchmark (step 7) remain a separate
issue.
