Find moved fragments through the fan-out, not their old store

The fan-out chunk store implements the placement-aware store interface
but kept the trait's default get_fragment_at / put_fragment_at, which
ignore the resolved D server and route by the fragment's own index
(stores[index % n]). That is correct only while placement is the
identity mapping. Once a custodian relocates a fragment — a rebalance
evacuation or a reconstruction re-placement that repoints its committed
placement to a different D server — a read through the fan-out still
went to the fragment's old store. With no redundancy (the
single-fragment EcScheme::None case) the object became unreadable; with
Reed-Solomon the read silently recovered around the misplaced fragment,
masking the loss while eroding the redundancy margin until enough moves
dropped it below k.

The fan-out now routes get_fragment_at / put_fragment_at by the D
server the caller resolved from the placement record, matching the
id-indexed GrpcChunkStore fleet. An un-moved fragment carries the
identity placement (dserver == index) and routes to exactly the store
it used before; only a moved fragment diverges, landing on the store
its committed placement names.

Fixes #356
