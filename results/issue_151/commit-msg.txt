ci: make the Rust gate a requireable check; mark dco gating-ready

The Rust gate (fmt/clippy/build/test/cargo-deny/conformance/DST) ran on
every code PR but could not be enforced: the whole `ci` workflow was
path-filtered, and GitHub leaves a path-skipped workflow's checks
"pending" forever, so requiring it would wedge every docs-only PR. That
is why #125 deliberately left `rust` out of the required-checks set — at
the cost that a PR failing `cargo xtask ci` stays mergeable. DCO had the
mirror gap: dco.yml is correctly always-on, but nothing recorded it as a
designated required check, so it reported without gating despite
ADR-0003 §1 requiring a sign-off on every commit.

Stop skipping the workflow; skip only the heavy job. The `ci` workflow
now always triggers. A cheap `changes` job classifies the diff as code
vs. docs-only (same `gh api` idiom as dco.yml / require-issue.yml, no new
action dependency, mirroring the #125 docs paths), and `rust` is gated on
that — so the docs-only skip is preserved, just relocated from the
workflow trigger to the job. An always-runs `gate` job aggregates the
result: it passes when `rust` succeeded or was skipped (docs-only) and
fails otherwise, and it carries no path filter so it always reports.
`gate` is the job a maintainer adds to the required-checks set without
wedging docs-only PRs. The classifier fails safe — an indeterminate
changed-file list runs `rust` rather than silently skipping it.

dco.yml gains a header note designating its `dco` context as a required
check (no behavioural change). The branch-protection flip that adds
`gate` and `dco` to the required set is a maintainer admin action, not an
in-repo change; this lands the precondition that makes it non-wedging.

Fixes #151
