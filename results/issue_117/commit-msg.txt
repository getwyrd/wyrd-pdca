Add Tier-2 container integration test and D-server throughput bench

This is the final step (PR step 7) of the accepted Milestone 2 proposal
0004 (networked D servers). Steps 1-6 built and DST-validated the data
path in simulation; this step proves it against reality and makes the
throughput-scaling claim first measurable.

A new Tier-2 integration test stands up multiple real, networked gRPC D
servers in containers and drives an end-to-end S3-style write -> read
under rs(6,3), asserting the read is byte-identical to the write. Unlike
the in-process Tier-0/1 tests, it exercises real tonic transport, real
HTTP/2 framing, real prost (de)serialization, and real connection
lifecycle -- i.e. that the abstractions Tier-1 simulates match reality.
A `cargo xtask integration` task brings the cluster up under docker
compose, resolves each replica's host port, runs the test against the
live endpoints, and tears the cluster down. The test is gated: it is
ignored and no-ops without its endpoints, so the default `cargo test`
and the docs-skippable `cargo xtask ci` lane never need Docker. A
nightly (and on-demand) workflow runs it -- the first container job in
CI -- not a required per-PR status check.

A second benchmark, run by `cargo xtask bench`, measures aggregate
write/read throughput across D-server counts {1,3,9} over real
in-process tonic, so scaling with D-server count is visible on a laptop
and the target compiles in CI. The numbers are tracked for regression
visibility, not gated, matching the existing EC micro-bench.

No new external crate is added: orchestration uses the host's existing
docker compose rather than a test-harness dependency, so the dependency
audit is not triggered. The ChunkStore trait, the wyrd.v0 proto service,
and the commit protocol are unchanged; this is wiring only.
