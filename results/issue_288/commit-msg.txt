core: schedule repair for fragments rejected as corrupt on read

A read that hits a fragment a verifying backend rejects as corrupt was
supposed to schedule that chunk for repair, the same way it does when it
decodes corrupt bytes itself. It only did so for the locally-decoded
case: when a real backend (the on-disk store, or the gRPC client mapping
a DATA_LOSS response) reported the corruption as a typed fault instead of
handing back the bad bytes, the read silently dropped the signal. For a
single-fragment object the read failed with nothing queued; for an
erasure-coded object the bad shard was treated as merely missing and the
object still read back, but the corruption was never recorded. Either
way the promised repair never happened, so the bad data sat unrepaired
until something else noticed it.

Both fragment-fetch sites on the read path now classify a corruption
fault the same way the scrubber does: record the chunk as a repair
obligation, then read around the fault where possible. A single-fragment
read records the chunk before surfacing the unavoidable error; an
erasure-coded read records the chunk and reconstructs from the surviving
shards. Transient (non-corruption) failures are left untouched and keep
propagating to the caller's retry policy, so they are never mistaken for
corruption.
