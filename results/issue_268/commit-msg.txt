Reconstruct remote dead-sector reads as block-read faults

When a fragment lives on a networked D server, a dead-sector read failure
(POSIX EIO) at the device used to arrive on the client indistinguishable
from a transient blip: the server flattened it to an INTERNAL status and
the client wrapped that as a transport error. Reconstruction then treated
a permanently unreadable fragment as something a retry might heal, so it
retried the same dead sector forever instead of reading around it and
rebuilding from the surviving fragments. Read-around already worked for
local fs-backed stores; it never reached the networked path, which is the
dominant production transport.

This makes a block-layer read fault keep its identity across the gRPC
wire. The server now maps a device read fault to a dedicated status code
(FAILED_PRECONDITION), separate from the DATA_LOSS code that carries
genuine corruption and from the transient codes; the client reconstructs
it into a new seam type, BlockReadFault. That type is permanent (so
reconstruction reads around it and rebuilds from the survivors) but is
explicitly NOT an integrity fault, so the scrub pass does not record a
dead sector as a corruption finding the way it does for true bit rot —
it takes the same branch a local EIO takes. A genuinely transient remote
failure still arrives as a transport error and keeps its retry behaviour.

The closure of "block-layer read fault" (errno 5 / EIO only) is now
defined once on the seam, so the server, the client, and the
reconstruction classifier agree without each re-deriving it. The local
and remote classification paths are left untouched.
