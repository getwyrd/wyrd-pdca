custodian: read around a placed fragment's permanent read fault

Before this change, a single bad disk could stall repair for an entire
cluster. When the custodian assessed a chunk for repair it fetched every
already-placed fragment and let any read error abort the whole pass. So one
D server with a dead sector — a block-layer EIO — would not just fail its
own fragment: it would halt reconciliation for every chunk sharing the
repair queue, and a disk that rotted after its data had landed could never
be repaired at all.

The assessment now classifies a fetch fault the same way the read path and
the scrub loop already do. A permanent durability fault — a corruption
fault, or a block-layer read fault where the device cannot return the bytes
— is read around: that fragment is treated as missing and the chunk is
rebuilt from its surviving shards, so one faulted server no longer blocks
the queue. A transient fault on a healthy server (unreachable, timed out,
busy) carries no durability signal, so it is still propagated to the retry
policy and never silently converted into a lost fragment or a needless
re-placement. The permanent-vs-transient distinction the durability seam
guarantees is preserved at this decision point, matching the behaviour of
scrub and the read path.
