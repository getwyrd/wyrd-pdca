fix(custodian): repair bit-rotten fragments instead of aborting scrub

A scrub pass over an on-disk or networked D server was meant to find a
bit-rotten fragment, queue its chunk for reconstruction, record it on the
durability metrics, and move on. Instead, the first corrupt fragment
aborted the entire pass: the store rejects a corrupt fragment on read, and
scrub propagated that rejection straight out of the loop before its repair
branch could run. So bit rot was never enqueued for repair, every fragment
after the corrupt one went unscrubbed, and the corruption/coverage metrics
stayed all-green while data silently decayed below its redundancy floor.

The miss was that scrub had no way to tell a *corruption* fault (bad bytes
that can never heal — repair and continue) from a *transient* one
(unreachable / timed out — propagate and retry); both arrived as the same
opaque error, so it had to treat them alike and bail.

This introduces a single typed integrity fault in the shared trait crate
that every backend produces and every consumer recognises. The on-disk
store surfaces a failed read-verify as that fault; the gRPC server carries
it across the wire as DATA_LOSS (distinct from a transient status) and the
client reconstructs the same type, so a networked store is classified
exactly like a local one. Scrub now branches on it: a corruption fault is
enqueued for reconstruction, counted, and the pass continues past it; a
transient fault still propagates to the retry policy. As a related fix on
the same classification seam, a client that PUTs a malformed fragment now
gets INVALID_ARGUMENT rather than INTERNAL, so it stops retrying bytes that
can never verify.
