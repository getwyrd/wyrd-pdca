fix(custodian): don't count aborted repairs as successes

The durability telemetry over-reported successful reconstruction
repairs. The dispatched-repair counter is incremented up front for
every plan a reconstruction pass attempts, and each non-success is
meant to be netted back out on its own counter so that "successful
repairs" derives as repaired minus the offsets. A repair that loses
its commit race was offset on the conflict counter, but a repair that
could not proceed at all (the placement selector picked a server
outside the fleet's view, so nothing was committed) was offset by
nothing. The success identity therefore over-counted true repairs by
the number of aborted plans, and the durability plane reported more
chunks repaired than were actually committed.

Add a distinct reconstruction_aborted counter and offset the aborted
outcome on it, mirroring the existing conflict counter, so the
identity becomes repaired minus conflict minus aborted. The
metric that matters — the up-front dispatched-repair count — keeps
being emitted before the heavy decode/commit section, where the OTel
bridge will not drop it under load; only the new offset is emitted
late, degrading exactly like the already-accepted conflict offset if
ever dropped. The in-code documented identity is updated to match.

This accounting first shipped in commit
5fb905c1eba54d55a5bec796972586bcd972fdde, which added the up-front
emission without an offset for the aborted outcome.
