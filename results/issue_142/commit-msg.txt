custodian: reclaim orphaned and expired-lease fragment bytes

Garbage produced by failed fan-out and by deletes/completed reconstructions
piled up unreclaimed: the only GC was a test-invoked stand-in that removed
expired `pending:` ledger entries but explicitly left the fragment *bytes*
behind, so storage was never actually reclaimed. The fenced reconciliation
control point stood up by 40c34130ac27e91a2a5b91d957e8d014fb7fb5f5 only
reported a satisfied zone and dispatched no maintenance work.

This hangs the first running maintenance loop — GC — off that control point.
`reconcile_step` now dispatches to a GC reconciler that reclaims both garbage
classes through the chunk store: the bytes behind an expired pending lease,
and orphaned fragments that no committed chunk map references. There is one
production entry — GC is the code `reconcile_step` runs, not a parallel
test-only path — so a custodian runtime, once it exists, drives this loop
unchanged.

Two invariants gate every reclaim. A fragment a committed chunk map's
placement record points at is never deleted (reclaiming it would be silent
corruption over data the request plane reports as live). And an orphan is
reclaimed only after a reader-safe grace window has elapsed past the instant
it was stranded, so an in-flight reader holding the prior version is never
torn; the window length stays a caller-derived value, not a baked-in constant.
Every reclaim and skip is emitted on the existing durability-plane telemetry
seam as a metric plus an append-only audit event.

No running custodian process drives the loop yet — that host is a later,
cross-cutting slice for all four loops. This change makes GC correct over the
metadata- and chunk-store seams and reachable through the real control point.

Fixes #142
