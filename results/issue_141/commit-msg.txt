custodian: place chunk fragments across distinct failure domains

Until now a chunk's n fragments were placed by M2's stateless index % n
route, which is blind to failure domains: two of a chunk's fragments
could land in the same rack, power feed, or switch, so a single
failure-domain loss could take both and RS(6,3)'s durability math was
not actually claimable within a zone.

This stands up the M3 custodian plane and makes placement
failure-domain-aware end to end:

  - a single active, fenced custodian, elected through the existing
    Coordination::elect_leader seam, whose monotonic fencing token
    rejects a superseded leader's actions;
  - a thin, zone-local failure-domain selector in core that spreads a
    chunk's n fragments across n distinct opaque domains wherever the
    topology offers >= n of them, and refuses (errors) rather than
    collide domains when it cannot;
  - the production write path rewired so a D server's registration
    carries an opaque failure-domain label and the committed placement
    record reflects the selector's distinct-domain choice instead of
    the identity index % n vector; the read resolves fragments from
    that record;
  - a backend-agnostic OpenTelemetry seam emitting the first custodian
    metric over both a Prometheus registry and OTLP push, with no
    backend hardcoded.

This builds on the placement record and stable DServerId added for
#139, which left the write at the identity index % n route and added no
registration failure-domain label; both of those are retired here.

The custodian crate depends only on the traits/core seams (plus the
tracing/OpenTelemetry stack), never a concrete backend.

Fixes #141
