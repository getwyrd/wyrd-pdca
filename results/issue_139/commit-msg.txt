Record fragment placement so a moved fragment is still found

Until now a read located each fragment by recomputing M2's stateless
`index % n` route, so it found a fragment only because nothing had moved
it. The moment a custodian relocates a fragment — the whole point of
Milestone 3 — that arithmetic resolves to the wrong D server and the read
fails. The committed chunk map now records, per fragment index, the
stable D-server id holding that fragment, written at the write commit
point; the read path resolves each fragment from that record instead of
`index % n`, so a relocated fragment is still read correctly, including
after a process restart.

The placement field is additive metadata on a never-yet-deployed schema
(`#[serde(default)]`): an inode written before it decodes with an empty
vector and falls back to the per-index identity placement, so M0-M2 data
reads through the same path. A new `PlacementChunkStore` seam carries the
address-by-stable-id capability with identity defaults, so a single store
or the `index % n` fan-out — each already its own location authority —
behaves exactly as before; only a genuinely relocatable fleet overrides
it.

This is the first slice of accepted proposal 0005 (Milestone 3 —
custodians): record-at-write and resolve-on-read. The version-conditional
location update used by repair/rebalance and the custodian-aware fan-out
are later slices.
