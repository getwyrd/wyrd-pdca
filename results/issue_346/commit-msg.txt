custodian: evacuate pre-M3 chunks when draining a D server

When an operator drains a D server to decommission it, rebalance
evacuation skipped any chunk written before the placement field shipped
(a pre-M3 / mixed-era record, whose placement vector decodes empty). The
chunk's live fragment was left on the draining server and lost once the
server was removed.

Rebalance planning scanned the raw placement vector instead of resolving
fragment locations through the identity-placement fallback that the read
path, GC, scrub, and reconstruction already share (ChunkRef::placed_dserver).
An empty vector produced no evacuation candidates, so the chunk was
silently skipped; the same raw vector was also carried into the commit, so
even a forced move would have panicked on the empty vector or persisted a
malformed, too-short placement record.

Resolve the full fragment set once through that shared fallback and reuse
it for evacuation selection, survivor-domain spread, and the placement
record committed back. A mixed-era fragment on a draining server is now
moved to a healthy, distinct-domain server, and the committed placement is
always full-length. This completes the rebalance wiring deferred by
093732d3e632f54f51859d527e042aafcc6447ae, which added the shared
placement resolution but left this caller on the raw vector.

Fixes #346
