fix(chunkstore-fs): unique write scratch ends same-id put race

Concurrent writes of the same fragment to the filesystem chunk store could
fail spuriously. Each write staged its bytes in a scratch file named only
after the fragment id, then atomically renamed it into place. Two writes of
the same fragment — an idempotent client retry, or a repair pass writing a
fragment a foreground write was also storing — therefore shared one scratch
file: their writes could clobber each other, and the second rename could hit
"No such file or directory", failing a legitimate write even though the bytes
were identical and a complete fragment had already been published.

Give each write its own scratch file, unique within the store, and keep the
atomic rename as the sole publish point. Concurrent writes of the same
fragment can no longer observe or overwrite each other's partial bytes, and
none fails spuriously; last-writer-wins is a semantic no-op because same id
means identical, checksum-verified bytes. Writes of different fragments stay
independent, exactly as before.

Unique scratch names no longer self-clean the way a single fixed name did
(the next write of the same id used to overwrite it), so scratch left behind
by a crash is now reaped when the store is opened. Only one process owns a
store's root, and no write is in flight at open, so reaping can never remove a
live write's scratch. Files ending in .frag are untouched, so reads and
fragment listing are unaffected.
