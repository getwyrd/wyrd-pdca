custodian: rebalance loop drains and decommissions D servers

An operator can now mark a D server draining or decommissioning and have
the custodian proactively evacuate its fragments, so a planned server
removal preserves durability the way an unplanned loss (reconstruction)
already does. Until now the custodian had GC, scrub, and reconstruction
hung off the fenced reconcile_step seam but no rebalance loop and no
declarative drain/decommission surface, so retiring a server risked the
chunk's failure-domain spread.

Management is declarative: writing desired state for a server is the
"policy changed" moment, and the rebalance loop reconciles reality
toward it until "policy satisfied" — the drained server holds no
referenced fragment. Each evacuation reuses the same commit-point-atomic,
version-conditional re-place as a reconstruction (the machinery that
landed with the slice-6 reconstruction work, #144): the fragment is
copied to a healthy server in a distinct failure domain first, then one
version-conditional commit repoints the placement record and orphans the
displaced copy. A crash mid-move leaves only collectable garbage, never a
torn chunk, and a racing writer loses the commit rather than corrupting
the record. Where a move cannot keep the chunk on n distinct domains,
spread wins and the move is refused.

The capacity plane now also emits per-failure-domain utilization on the
durability telemetry seam. The operator-facing management API and CLI
stay deferred to ADR-0013; desired state folds into local metadata,
single-zone.
