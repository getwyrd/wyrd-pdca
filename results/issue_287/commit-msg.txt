Fix GC deleting live fragments of pre-M3 committed objects

Garbage collection could permanently delete fragments of a committed
object whose chunk map predates M3 placement records. The read path
resolves such a fragment by identity fallback — fragment index i lives on
D-server i when the placement vector is empty or short — but GC's safety
reference set only walked the explicit placement vector. For a pre-M3
object that vector is empty, so GC protected nothing, and a stale orphan
record or an expired pending-ledger entry pointing at the same fragment
let GC reclaim live, readable data. Scrub shares the same reference-set
computation and so shared the gap.

GC's protected set now equals the placement closure the read path
actually resolves: it expands every committed chunk to its full fragment
count (1 for EcScheme::None, k+m for ReedSolomon) and applies the same
identity fallback for missing placement entries. A fragment a committed
chunk-map reference resolves to is therefore never passed to
delete_fragment.

The identity-fallback resolution and the scheme-aware fragment count are
factored into two ChunkRef methods (placed_dserver, fragment_count) so
the read path, reconstruction, scrub, and GC share one definition and
placement semantics cannot drift between them. The read-side fallback was
introduced alongside the placement record in
093732d3e632f54f51859d527e042aafcc6447ae; the GC reference set added in
af4ab65aac652d495c124961d6f9586e7a7e7999 was never updated to match, which
this change corrects.

Fixes #287
