test(dst): add the Tier-0 custodian property campaign

M3's four custodian loops — GC (#142), scrub (#143), reconstruction
(#144) and rebalance (#145) — each shipped with their own per-slice
tests, but the consolidated verification campaign that is M3's
graduation gate did not exist. This adds it, so every custodian
correctness property is now machine-checked continuously inside the
deterministic simulator and reproduces from its seed.

The new dst integration test (crates/dst/tests/custodian.rs) sweeps the
six proposal-0005 properties across 50 madsim seeds, each driven through
the real reconcile_step fenced control point over in-memory trait
stores: reconstruct-to-full-redundancy in a distinct failure domain (Q1),
commit-point-atomic repair under crash (never a hybrid chunk), scrub
detecting bit-rot then reconstructing around the excluded shard (Q2), GC
reclaiming only true orphans (Q3), a fenced stale leader landing no
update, and durability-plane emission rising then returning to zero. A
fixed REGRESSION_SEEDS set replays all six per run so any future
bug-finding seed becomes a permanent regression (ADR-0009).

To drive those faults from the run seed the testkit gains a storage
fault seam — StorageFault plus SeededStorageFaults, the bit-rot /
fragment-loss and D-server-kill sibling of the existing net seam — and
xtask gains the deferred Tier-1/Tier-2 runners (disk-faults, jepsen,
kill-reconstruct) which skip cleanly unless explicitly opted in, so they
stay out of the unprivileged ci gate. No production custodian behaviour
changes; this slice only verifies the loops the earlier slices built.

Fixes #146
