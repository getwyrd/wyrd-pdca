test: pin mixed-era placement resolution across all consumers

A chunk's fragment locations resolve through one authoritative
identity-placement fallback: when a record's `placement` vector is empty or
shorter than the fragment count — the shape that pre-M3 / mixed-era objects
carry — fragment `i` resolves to D-server `i`. The read path, GC, scrub,
reconstruction, and rebalance all rely on this, and recent fixes restored it for
objects written before the placement field existed (GC reclamation in
9d0af207b5b4fd60d35c13388afef1bc65eb47ab, draining evacuation in
04ad01aab4bdeefe29e429f84610abf1b963e1d3).

The regression coverage for that resolution was uneven: most consumers were
tested only against full-length placement vectors, where the fallback is never
exercised. Two behaviours that are correct today — rebalance evacuating a pre-M3
chunk, and reconstruction re-placing one back to a full-length record — had no
test pinning them, so a consumer could silently regress to reading the raw
vector without any check failing.

This adds the missing matrix: every placement consumer (read, GC, scrub,
reconstruction, rebalance) now has explicit coverage for empty, short, and full
placement across the no-erasure and Reed-Solomon schemes, including a
Reed-Solomon {6,3} case per consumer; a deterministic-simulation scenario seeded
with an empty-placement chunk asserting that read and maintenance resolve it
identically across seeds; and a reconstruction case proving an empty-placement
chunk is re-committed as a full-length record with the rebuilt fragment in a
distinct domain. The change is test-only; no production code is touched.
