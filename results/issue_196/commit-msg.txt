Build the Tier-2 kill-and-reconstruct durability harness in xtask

The M3 verification campaign promises a Tier-2 leg: on a single real node,
kill a live D server and watch the production custodian rebuild the lost
fragments over a real gRPC fleet. That leg was never built. `cargo xtask
kill-reconstruct` only shelled out to `WYRD_TIER2_CMD`, an environment
variable that is not defined anywhere in the repository, so the campaign
ran nothing and no test exercised the kill-and-reconstruct path.

This replaces the external-command shim with a real in-repo harness. A new
`#[ignore]`d scenario test stands up a ten-server RS(6,3) cluster under the
existing docker-compose plumbing, kills one D server, drives the production
reconstruction path (`reconcile_step` -> `reconstruction::reconcile`)
against the live containers, and asserts three durability properties: the
victim's fragments return to full redundancy in distinct failure domains; a
crash before the version-conditional commit leaves collectable garbage on
the spare and a fully-old inode, never a torn chunk; and the repaired
fragments still erasure-decode to the original bytes.

The harness's host-independent logic (kill-victim selection, container
naming, and the three durability assertions) is covered by unit tests that
run inside `cargo xtask ci`, so a regression that stubbed the harness fails
the gate at Check. The live container run stays out of the unprivileged
gate and is exercised by a new scheduled, opt-in CI job.

Fixes #196
