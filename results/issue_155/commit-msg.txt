Add gateway client mode to run a local cluster from the CLI

Until now there was no way to drive Wyrd's distributed path by hand: `wyrd
put`/`get` only talked to a local disk, and the one multi-container compose
was a CI fixture on ephemeral ports that `cargo xtask integration` scaled and
tore down. You could not stand the system up on one machine and poke it.

This adds an `--endpoints` mode to `put`/`get` that fans each object's
erasure-coded fragments out over gRPC to a configured list of networked D
servers, holding the object metadata locally under `--data-dir`. A user-facing
`docker-compose.yml` (distinct from the test fixture) brings up four D servers
on fixed published ports, each with its own persistent volume, and a README
"Run a local cluster" walk-through ties it together.

The cluster path deliberately reuses the local-disk path's persisted id
machinery — `alloc_inode` over the `meta:next_inode` counter plus
inode-derived chunk ids — instead of the Gateway struct's in-process counters,
which reset every process. That is what lets you store several distinct
objects across separate `wyrd` invocations under one `--data-dir`: with the
in-process counters the second PUT re-allocated inode 1 and failed as a
spurious "concurrent writer won" conflict, and reused chunk id 1 — silently
overwriting the first object's fragments on the shared D servers.

Four D servers, not three: the default rs(6,3) places 9 fragments per chunk
needing any k=6, so a three-server topology drops to exactly k on a single
loss; a fourth keeps redundancy headroom, matching the loopback round-trip
test.

Dynamic discovery (etcd / non-static coordination), stable placement, and
rebalance remain milestone-3 work; this is the static-endpoints, single-gateway
on-ramp only — no change to the ChunkStore trait, proto, or commit protocol.
